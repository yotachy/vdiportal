# 회고 존 (Retro Zone) — 오류 귀속 기반 지표 재조합 진단 + 승격 게이트 — 설계

날짜: 2026-07-14 · 상태: 승인됨(사용자) · 범위: 1차 스펙(백엔드 회고 존 + 게이트). R3(사용자 선택 심화 UI)는 2차 스펙으로 분리.

## 배경 / 문제

절대 방향 예측은 9번째 벽에 도달했다 — 검증 6축을 스태킹해도 "항상상승" 베이스라인 대비 BSS≈0(정보량=베이스레이트 상수 동급). 다수의 신규 축(인트라데이·매크로·인터마켓·거래량·왜도·breadth·8-K)이 "가격 재조합=새 정보 0"으로 기각됐다. 새 *예측 축*의 한계효용이 급감한 상태.

기존 개선 방식은 **후보 축을 하나씩 관문에 던지는 하향식(top-down) 탐색**이다. 이 스펙은 상보적인 **상향식(bottom-up) 탐색**을 도입한다: 엔진이 *어디서 체계적으로 틀리는지*를 회고 분석하고, 그 오류 구조에서 **지표 재조합**을 발굴한다. 단, 대부분의 발굴은 같은 벽("가격 재조합=0")에 막힐 것이므로 **정직한 관문과 격리가 설계의 핵심**이다.

사용자 요구(3겹):
1. **R1** — 회고분석은 백테스트 범위 안의 **격리된 관리·제어 영역**이어야 한다(숨은 자동루프 아님).
2. **R2** — 회고에서 나온 것 중 **개선된(관문 통과) 것만** 분석엔진(forge-core)에 반영하는 **단방향 절차**.
3. **R3(심화)** — 회고가 *큰 도출점·예측방향 기준*을 **제시** → 사용자 **선택** → 분석이 **달라지고** → **무엇이 바뀌었는지 가시화**. (2차 스펙)

## 결정 요약(브레인스토밍 합의)

- **기준의 형태 = 지표 재조합 추천**(사용자 선택). 신뢰도 메타모델은 주 산출물에서 빠지고 진단의 *근거 요소*로만 편입.
- **발굴 방식 = ② 오류 귀속·지표 진단**. 이유: (a) R3가 요구하는 "왜 바뀌었나"를 유일하게 자기설명, (b) 자유도가 낮아 스누핑 저항이 높고 관문 통과분이 실제로 나옴, (c) "큰 도출점"("이 국면에서 지표 Z가 배신")이 자연 산출.
- **계승 구조**: 이 스펙(② 발굴 + 엄격 OOS/LOSO)으로 가설을 뽑고, **③ 챔피언/도전자 walk-forward 최종검증은 3차 라이브 존**으로 미룬다.

## 공통 원칙 (사전 등록 — 과장·스누핑 차단)

기존 백테스트 하네스 규율을 그대로 계승([[scoopforge-backtest]] · direction-redefine 선례):

- 채택 관문(실험 **전에** 고정): OOS(시간분할) 순증분 **≥1.0pp**(방향류 기본선; 상대강도 축은 +1.5pp 선례) · 전반부/후반부 분할 **양쪽 양수** · 모델형은 종목외 **LOSO** 유지 · **자명규칙 베이스라인**(다수결·지속성·±지속성·±모멘텀·안티드리프트) 대비 우위.
- **정직 판정이 기본값**: 대부분의 재조합 후보는 "개선 없음 — 현 조합 유지"로 판정될 것이며 그 null도 대장에 기록한다("이 재조합은 실측상 도움 안 됨"도 정보). 채택은 소수.
- **스누핑 방지 핵심**: 회고가 사용자에게 보여주는 "적용 시 예측 변동/개선"은 **반드시 OOS/LOSO 근거**로 표기한다. in-sample 개선 표시 금지(사용자가 스누핑을 opt-in하게 됨). R2의 게이트가 R3(사용자 제시)에도 동일하게 걸린다 — 엔진 승격이든 사용자 제시든 **같은 관문**.
- 하네스 규약: `map/backtest/` 결정론(`Date.now`/`Math.random` 금지·스탬프 주입), lookahead 금지, **배포 제외**, 실패도 스코어카드 이력 기록([[scoopforge-scorecard-changelog]]).
- **저자유도 규율**: 진단 1건 → 재조합 변경 1개(그 국면에서 지표 제거·감가 또는 누락 지표 추가). 자유 가중 탐색 금지(①의 신기루 지뢰밭 회피).

## 아키텍처

### A. 격리·위치 (R1)

신규 격리 영역 `map/backtest/retro/`. 기존 랩과 동형(결정론 node 스크립트). **forge-core에는 게이트 통과 전까지 아무것도 닿지 않는다.** 실행 가능·범위 지정(티커/기간/국면)·검사 가능한 "관리 존". 배포 제외.

### B. 5단 파이프라인

```
1. 오류 원장 빌더 (miss-ledger.js)
   엔진을 히스토리에 재생 → 예측건별 레코드:
   { sym, tf, t, regime[], perIndicator:[{id, driftSign, mag, conf}],
     pred(dir/prob), outcome(correct/wrong), errMag }
   (stack-records.json 선례 — 엔진 1패스 캐시 재사용, 재계산 회피)
        ▼
2. 귀속 엔진 (attribution.js)
   미스마다 지표 책임도 산정:
   - 반대 신호: 지표가 오답 방향을 가리켰나
   - 확신 오류: 높은 conf로 틀렸나
   - 누락 신호: 빠진 지표 W를 넣었으면 잡았을까(반사실 1개 지표 add 재생)
   국면 버킷 집계 → 진단 카드:
   "지표 Z가 국면 X 미스의 N%에서 배신 (base 대비 편향 ΔΔ, 표본 n)"
        ▼
3. 재조합 후보 (remix-candidates.js)
   진단 1건 → 구체 제안 1개(저자유도):
   { regime, change:{drop|downweight|add, indId}, rationale, sourceDiagId }
        ▼
4. 승격 게이트 (retro-gate.js)
   후보를 기존 관문에 그대로:
   OOS/LOSO · 자명규칙 · 순증분≥1.0pp · 전후반 부호 · brierDecomp/BSS(metrics.js)
   판정: 채택 | 개선없음 | 표본부족  + oosDelta(정직 근거)
        ▼
5. 회고 대장 (retro-catalog.json)
   내구 산출물 — 진단 + 게이트 판정 + OOS 델타의 카탈로그.
   R3 UI가 읽는 단일 출처. validatedAxes의 회고판.
```

### C. 단방향 승격 (R2)

회고 대장에서 **판정=채택 + 근거 충분**인 항목만 forge-core 반영 *후보*가 된다. 흐름은 단방향·명시적 — **엔진은 자동 변이하지 않는다**. 반영은 기존 축 채택처럼 의도적 한 걸음: forge-core에 **국면조건부 드리프트 조정**(해당 국면에서 지표 드리프트 계수 스케일)으로 들어가거나, 충분히 강하면 **새 검증축**으로 승격되어 `validatedAxes` 레지스트리에 등록(→ 스코어카드·UI 자동 확장, 열린 엔진 원칙 [[scoopforge-open-engine-principle]]).

### D. 회고 대장 스키마 (R3가 바로 먹을 수 있게 1차에서 확정)

```js
retroCatalog = [{
  id,                       // uid
  diagnosis: {              // "큰 도출점"(사람이 읽는 진단)
    regime,                 // 국면 태그(추세/변동성/이벤트근접 버킷)
    indicator,              // 배신/누락 지표 id
    kind: "betray"|"overconfident"|"missing",
    stat: { missShare, baseBias, n }
  },
  remix: {                  // 사용자가 토글할 재조합
    change: { op:"drop"|"downweight"|"add", indId, factor? },
    rationale             // 자연어 근거(기전 설명)
  },
  verdict: "adopt"|"no-improvement"|"insufficient-sample",
  evidence: {               // 정직 표기(in-sample 금지)
    oosDelta, loso, halves:[h1,h2], baselineBeat, bss, n, gate:"pass"|"fail"
  },
  promoted: bool            // forge-core 반영 여부
}]
```

### E. R3(사용자 선택 심화) — 2차 스펙, 여기선 계약만 고정

회고 대장(게이트 통과분) → 사용자에게 카탈로그 제시 → 토글 → 분석 재실행(국면조건부 재조합) → **before/after 가시화**: 방향·확률·콘 변화 + **OOS 근거 병기** + "국면 X에서 지표 Z 배신" 기전. 1차의 대장 스키마(D)가 이 UI의 데이터 계약이다.

### F. 메타모델(신뢰도)의 자리

주 산출물에서 제외. 진단의 근거 요소("이 국면에선 신뢰도 낮음")로만 편입. 별도 축이 필요하면 향후 별개 스펙.

## 비목표 (YAGNI)

- 자유 가중 최적화·국면 버킷 브루트 탐색(①) — 스누핑 위험으로 배제.
- walk-forward 챔피언/도전자(③) — 3차 라이브 존으로 분리.
- R3 사용자 선택 UI·before/after 가시화 — 2차 스펙.
- 신뢰도 메타모델을 독립 축으로 — 향후 별개 판단.
- forge-core 실제 반영(승격 실행) — 대장에 채택 후보가 실제로 나온 뒤 별도 커밋.

## 리스크 / 완화

- **대부분 "개선 없음"으로 끝날 위험(가장 큰 리스크)**: 예상된 결과. null도 대장에 남겨 "정직 노선" 자산으로. 채택 0건이어도 파이프라인·귀속 진단 자체가 재사용 인프라.
- **국면 버킷 표본 부족**: 버킷은 소수(추세/변동성/이벤트근접)로 시작, 표본<임계는 "insufficient-sample" 판정. LOSO 필수.
- **반사실(누락 지표 add) 비용**: 지표 1개 add 재생은 stack-records 캐시 위에서 증분 계산. 지표 전조합 폭발 금지(add 후보는 진단이 지목한 것만).

|---|---|---|
| **1차(본 문서)** | 회고 존 + 오류귀속 5단 + 단방향 게이트 + 정직 대장 | 설계 승인 → 구현 계획 |
